# Vibe Coding Core — G / O / M 框架定义

> 适用项目：Distill Chrome Extension

---

## 角色定义

### 生成器 G（Generator）
**职责**：根据当前实施步骤的意图，生成初始提示词 P₀  
**输入**：implementation-plan.md 中的某个 Step 描述 + 当前项目上下文  
**输出**：结构化的初始提示词 P₀  
**规则文件**：`g-updated-rules.md`（M 持续迭代更新）

### 优化器 O（Optimizer）
**职责**：按标准 Ω 对 P₀ 进行优化，输出最终提示词 P  
**标准 Ω**：
1. **精确性** — 明确指定文件路径、函数名、接口名，不用模糊词
2. **完整性** — 包含所有约束条件（禁止做什么、必须遵守什么）
3. **可验证性** — 每个提示词必须包含明确的验证标准
4. **模块化** — 强调单文件单职责，禁止在一个文件里写超过 200 行
5. **上下文锚定** — 提示词必须引用 memory-bank 中的相关文档
6. **防退化** — 明确说明不能改动哪些已完成的文件

### 元生成器 M（Meta-generator）
**职责**：分析每次 O 的优化过程，提炼出规律，更新 G 的规则  
**触发时机**：每完成一个 Phase 后  
**更新目标**：`g-updated-rules.md`

---

## 优化标准 Ω — 详细检查清单

执行 O 时，对 P₀ 逐项检查：

| 检查项 | 问题 |
|--------|------|
| 文件路径 | 是否明确指定了要创建/修改的文件的完整路径？ |
| 接口引用 | 是否引用了 `src/types/index.ts` 中定义的类型？ |
| 依赖说明 | 是否说明了该 Step 依赖哪些已完成的 Step？ |
| 禁止事项 | 是否明确了不能做什么（如：不能修改已完成的文件）？ |
| 验证标准 | 是否有具体的、可操作的验证方法？ |
| 技术栈一致 | 是否与 tech-stack.md 中的技术选型一致？ |
| 无代码泄露 | 提示词本身不包含实现代码，只写意图和约束 |
| 中文输出 | 注释和文档用中文，代码标识符用英文 |
